Clave — Amber-Parity Roadmap

Tiered feature design specs derived from the Amber vs. Clave comparison

2026-05-22 · iOS NIP-46 remote signer

Amber's same-device NIP-55 intents remain impossible on iOS — out of scope by platform. These four tiers close the other major gaps. Tiers 1–3 are specced; Tier 4 is flagged as exploration only. Tier 1 (key backup) is the priority: it's table stakes for a custody app and Clave's biggest exposure vs. Amber today.

Tier 1 Priority Table stakes

Key Backup & Recovery

Gap vs Amber
Clave holds the only copy of the key (Keychain, ThisDeviceOnly). Only export is raw nsec → clipboard. No seed words, no encrypted backup, no QR, no restore. Lose the phone → identity gone forever.
Solution
Four layered capabilities: (1) NIP-49 encrypted export (ncryptsec), (2) NIP-06 seed-phrase backup & restore for newly-generated keys, (3) QR export of the encrypted blob, (4) opt-in encrypted iCloud backup. Plus a backup prompt in onboarding.
NIPs
NIP-49 · ncryptsec NIP-06 · seed words NIP-41 · migration (future) FROST · sharding (future)
Key decision
Option B — encrypted blob, not iCloud Keychain sync. iCloud stores only the passphrase-encrypted ncryptsec; the plaintext key never leaves the device, so the security pitch holds. Working key stays ThisDeviceOnly — backup is additive and never touches the NSE signing path. New keys generated from a BIP-39 mnemonic (m/44'/1237'/0'/0/0) so seed words exist; legacy/raw keys get ncryptsec+iCloud only. Reuse rust-nostr for NIP-49/06 rather than hand-rolling scrypt + XChaCha20-Poly1305.
Tier 2 Low effort · high trust

App-Launch Lock

Gap vs Amber
Auth gates only key reveal + the notification-banner Approve. The app opens freely and the in-app approval sheet isn't gated — an unlocked phone exposes accounts, activity, pairings, and lets anyone pair/approve. Amber gates the app.
Solution
Optional Face ID / Touch ID / passcode lock on launch & resume (configurable timeout); gate the in-app approval commit; opaque privacy overlay over the app-switcher snapshot.
NIPs
None — iOS LocalAuthentication
Key decision
A UI access gate, not at-rest encryption (the Keychain/Secure Enclave already does that). System auth sheet, no custom PIN. NSE signing keeps working while the app is locked — the lock never blocks push signing.
Tier 3 Reliability

Multi-Relay Bunker

Gap vs Amber
The bunker:// flow pins every client to one relay (relay.powr.build) because the proxy subscribes to exactly one. Relay restart / rate-limit / dropped event → signing silently fails. Single point of failure.
Solution
Multi-relay end-to-end: bunker URIs advertise a relay set; proxy subscribes to all with cross-relay dedup; responses publish back to the originating relay (and the rest of the set). Tolerates partial relay outage.
NIPs
NIP-46 · remote signing
Key decision
Spans proxy (Node.js) + iOS NSE; sequence proxy-first, backward-compatible (RELAY_URLS defaults to today's single relay). Dedup by event id is the core correctness mechanism. Keep relay.powr.build in the default set so existing paired clients work with no re-pair.
Tier 4 Exploration — not specced

Tor / Privacy Routing

Gap vs Amber
Amber routes through Orbot so the signer never touches a clearnet relay. Clave's proxy sees token↔pubkey pairings and relays see the NSE's IP.
Why deferred
No Orbot equivalent on iOS; would require an embedded Tor or onion-routing the proxy/relay hop — materially harder, lower priority than backup/lock/reliability. Captured as a roadmap note, not a full design.
